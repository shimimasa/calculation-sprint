1) 結果画面に「Lv / XP」表示（表示のみ）

既存の結果サマリーUIに追記するだけ

まだ成長効果を付けなくても成立

2) XP/Level を profile 単位で保存（localStorage）

dashStatsStore/dashSettingsStore と同系統で追加しやすい

スキーマ増えるがロジックは単純

3) XP付与ルール（正解+1、ボス撃破+10、完走+5 など）

既存カウンタ（correctCount等）や run終了タイミングにフックするだけ

減点なしにすると分岐が減る

4) レベルアップ演出（テキスト/SE程度）

“強くなった感”を低工数で作れる

まずはUIテキストだけでOK

5) 敵の上に「名前＋HP」表示（通常HP=1固定）

既に「敵の上に情報」を検討している流れに乗る

通常敵はHP=1で固定にすると工数が跳ねない

6) ダメージ判定を明確化（正解→HP-1→撃破演出）

「正解時に敵を倒す」ルートが明確になる

ただし敵状態管理（approaching/defeated等）に触れるので少し上がる

7) 視認性改善（走者・敵を少し拡大、ヒット演出強化）

CSS/スプライトサイズ/当たり判定の再調整が発生しやすい

見た目だけで済ませる範囲に絞れば中工数

8) “武器っぽい演出”を追加（見た目だけ）

装備システムは作らず、斬撃/エフェクト/SEで満足度を取る

アセット追加が絡むと工数が増える（ただしロジックは軽い）

9) ボス導入（HP=2〜3、登場/撃破演出）

スポーン・HP・演出・図鑑/称号との接続点が増える

ここから設計コストが上がる

10) 成長効果（壊れにくい範囲）

正解タイム加算 +α（上限あり）

被弾ペナルティ軽減（上限あり）

ボスへの追加ダメージ解放
→ 数値調整・モード差分が絡むので中〜高工数

11) 速度成長（上限/条件付き/相殺込み）

難易度とテンポが壊れやすく、調整・検証が増える

同じ成長でも最後に回すべき枠

12) 図鑑（遭遇＝登録）

保存・一覧UI・アイコン整備が必要

“最初は埋まるだけ”でもUI工数が重い

13) 称号

条件設計＋表示＋保存が必要

図鑑より軽い場合もあるが、要件が膨らみやすい
